# V6.6.1 Hotfix 问题与修复日志

更新时间：2026-06-18  
版本：`0.6.6.1-hotfix`（`versionCode 13`）  
设备：红米 K60（设备代号 `mondrian`，ADB 设备 `b32a4e4`）

## 1. 问题现象

V6.6 Test Release 使用过程中出现近期消费记录未进入 App 的严重问题。系统通知监听权限仍显示已授权，自动监听开关也可能显示已开启，但实际付款通知没有进入 `NotificationListenerService.onNotificationPosted()`，属于“界面看起来正常、监听链路实际失效”的风险场景。

## 2. 排查结果

1. 自动监听开关保存在 SharedPreferences：`auto_listener/enabled`。
2. `KeepAliveNotificationService` 使用 `startForegroundService()` 启动，并通过 `ActivityManager` 查询真实运行状态。
3. 旧健康判断主要依赖通知监听权限、`listenerConnected` 和 90 秒心跳。
4. 心跳由 `PaymentNotificationListenerService` 自身协程更新，只能证明服务对象仍在运行，不能证明系统仍在向 `onNotificationPosted()` 分发通知。
5. 旧的一键修复在调用 `requestRebind()` 后仍以 connected/心跳判断结果，存在重绑后继续误报 healthy 的可能。
6. V6.6 双 Tab UI 没有关闭监听服务，但页面刷新无法弥补上述“假健康”判断缺口。

## 3. 最可能原因

Android/MIUI 在长时间后台运行、应用更新、系统省电或通知监听绑定异常后，可能保留通知访问授权和服务对象，但实际通知回调链路已失效。旧版没有主动发送并确认捕获一条真实通知，因此无法及时识别这种状态。

## 4. 修复内容

- 新增低优先级专用 probe notification，标题为“监听状态检测”，发送后自动取消。
- probe 使用 token 关联发送与捕获，避免把其他通知误认为探测成功。
- `PaymentNotificationListenerService` 捕获 probe 后更新成功时间和健康状态，并记录 `probeSuccess`。
- probe 超时后记录 `probeFail`，状态改为 suspicious，不再仅凭心跳显示 healthy。
- 看门狗、前台服务启动和 App 回前台时执行真实探测；探测流程增加互斥，避免并发 probe 相互覆盖。
- probe 失败后才进入带冷却的 `requestRebind`；冷却时间保持 10 分钟。
- 仅前台服务缺失时不消耗 requestRebind 冷却，提示用户恢复自动监听。
- 【监听】页新增“监听急救”，逐步显示权限、前台服务、连接、probe、requestRebind 和最终结论。
- 自动监听已开启但前台服务未运行时，显示恢复按钮；恢复后立即执行健康检查和 probe。
- 后台诊断新增 probe、自动恢复、前台服务缺失及 probe 失败后重绑等计数。
- 未修改 Room 表结构，未清空或迁移账单数据，未修改付款通知解析逻辑。

## 8. 手动验证步骤

1. 打开 App，进入【监听】。
2. 确认自动监听为已开启、前台服务为运行中。
3. 点击“监听急救”，确认 probe 显示成功且最终结论为“监听正常”。
4. 让另一台设备发送微信普通消息，进入“真实通知测试”检查微信捕获结果。
5. 完成一笔真实微信或支付宝付款，检查通知调试日志和待确认账单。
6. 打开“诊断报告”，查看 `probeSuccess`、`probeFail` 和重绑统计。

## 9. 仍无法完全解决的系统限制

- Android/MIUI 可以随时回收或限制通知监听服务，App 无法保证永久不被系统杀死。
- `NotificationListenerService.requestRebind()` 只是恢复请求，系统不保证成功。
- 微信/支付宝是否生成包含金额和商户信息的系统通知由其自身版本、通知设置和付款场景决定。
- 用户关闭 App 通知权限后，App 无法发送 probe，只能提示恢复通知权限。
- 小米省电、自启动、后台运行及锁屏通知策略仍需用户手动允许。
